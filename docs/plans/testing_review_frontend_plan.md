# SortFlow 前端单测与代码评审计划

## 目标

1. 建立前端单元测试体系，覆盖关键组件与业务流程。
2. 对照 `frontend_plan.md` 评审实现一致性、结构与质量。
3. 形成可复用的前端 Review 模板与问题清单。

## 范围

- 核心 UI 组件与页面行为
- 状态管理与接口适配
- 业务流程（预览 → 执行 → 进度 → 历史）

## 测试策略

### 推荐框架

- Vitest + React Testing Library

### 分层策略

- 组件层：UI 输入/输出与交互
- 页面层：核心流程交互与状态变化
- API 层：mock 方式验证请求/响应适配

### 测试数据

- 使用 MSW 或简单 fetch mock 固定响应
- 对列表/分页等用例提供稳定的 fixture

### 覆盖重点

1. **文件列表/预览**：数据展示、冲突标识
2. **执行流程**：任务创建 → 进度显示 → 完成反馈
3. **历史列表与撤销**：分页、详情展示、撤销触发
4. **系统配置**：源目录/预设/目标根目录管理

## 代码评审策略

### Review 维度

1. **与方案一致性**：对照 `frontend_plan.md` 检查功能是否完整实现
2. **结构与职责**：组件职责是否清晰、是否存在重复逻辑
3. **错误处理与日志**：前端错误提示与日志输出是否一致
4. **安全与边界**：输入校验、路径/文件名展示安全性
5. **可测试性**：组件与逻辑是否易于 mock 与隔离

### Review 输出

- 对每个模块记录：
  - 是否符合方案
  - 主要问题与建议
  - 是否需要补充测试或重构

## Review 模板（建议）

### 结构与一致性

- 是否符合方案文档的功能描述
- 组件职责是否清晰、是否存在重复逻辑
- API 调用与响应处理是否一致

### 质量与安全

- 错误提示是否清晰、是否覆盖主要异常
- 日志是否包含关键信息且不泄漏敏感数据
- 用户输入是否经过必要校验

### 可测试性

- 是否能在不依赖外部环境下执行测试
- 是否便于 mock（例如 API 适配层）

## 里程碑建议

1. **M1**：完成测试框架基线与关键模块单测（P0）
2. **M2**：补齐 P1/P2 业务单测与基础 Review
3. **M3**：完成 Review 总结、修复任务拆分与复测
