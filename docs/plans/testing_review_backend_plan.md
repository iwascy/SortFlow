# SortFlow 后端单测与代码评审计划

## 目标

1. 建立后端单元测试体系，覆盖核心业务逻辑与安全边界。
2. 对照 `backend_plan.md` 评审实现一致性、结构与质量。
3. 形成可复用的后端 Review 模板与问题清单。

## 范围

- 业务核心逻辑：预览计算、执行引擎、历史回滚、配置管理
- 数据层：模型与数据库操作
- 边界与安全：路径校验、文件系统访问

## 测试策略

### 推荐框架

- `testing` + `testify`

### 分层策略

- service 层：以纯逻辑单测为主，使用临时目录与假数据
- pkg 工具：路径校验、缩略图缓存命中逻辑
- handler 层：可选使用 `httptest` 做轻量 API 测试

### 测试数据

- 文件系统与路径相关测试使用 `t.TempDir()` 创建临时目录
- 预览/执行引擎测试使用固定样例文件名与规则集
- DB 相关测试使用 sqlite 内存库或临时文件

### 覆盖重点

1. **预览计算引擎**：命名规则、批次冲突、磁盘冲突、自动后缀
2. **执行引擎**：移动/复制正确性、进度计算、错误记录
3. **历史服务**：写入/查询/撤销逻辑
4. **配置服务**：CRUD 与排序
5. **路径安全**：只允许白名单根目录

## 代码评审策略

### Review 维度

1. **与方案一致性**：对照 `backend_plan.md` 检查功能是否完整实现
2. **结构与职责**：层次划分是否清晰，避免 handler 过重
3. **错误处理与日志**：是否统一格式/日志记录齐全
4. **安全与边界**：路径校验与文件系统访问的安全性
5. **可测试性**：是否能被单测覆盖，是否便于注入依赖

### Review 输出

- 对每个模块记录：
  - 是否符合方案
  - 主要问题与建议
  - 是否需要补充测试或重构

## Review 模板（建议）

### 结构与一致性

- 是否符合方案文档的功能描述
- 模块职责是否清晰、是否存在重复逻辑
- API 路由/参数命名是否一致

### 质量与安全

- 错误处理是否统一、是否存在遗漏
- 日志是否包含关键信息且不泄漏敏感数据
- 路径/文件操作是否全部经过安全校验

### 可测试性

- 是否能在不依赖外部环境下执行测试
- 是否便于 mock（例如接口抽象、依赖注入）

## 里程碑建议

1. **M1**：完成测试框架基线与关键模块单测（P0）
2. **M2**：补齐 P1/P2 业务单测与基础 Review
3. **M3**：完成 Review 总结、修复任务拆分与复测
