# SortFlow 全项目单测与代码评审计划

## 目标

1. 建立覆盖后端与前端的单元测试体系，保障核心逻辑可回归、可扩展，并可在 CI 中自动执行。
2. 根据已有方案文档（backend_plan.md / frontend_plan.md）对实现进行系统性 Review，确保功能、结构与约定一致。
3. 形成可重复的 Review 模板与问题清单，便于后续迭代复用。
4. 使用任务清单（CSV）进行进度跟踪与验收。

## 范围

- **后端（Go + Gin + GORM）**
  - 业务核心逻辑：预览计算、执行引擎、历史回滚、配置管理
  - 数据层：模型与数据库操作
  - 边界与安全：路径校验、文件系统访问
- **前端（Vite + React/TS）**
  - 核心 UI 组件与页面行为
  - 状态管理与接口适配
  - 业务流程（预览 → 执行 → 进度 → 历史）

## 测试策略

### 后端单测

- **推荐框架**：testing + testify
- **分层策略**：
  - service 层：以纯逻辑单测为主，使用临时目录与假数据
  - pkg 工具：路径校验、缩略图缓存命中逻辑
  - handler 层：可选使用 httptest 做轻量 API 测试
- **测试数据**：
  - 文件系统与路径相关测试使用 `t.TempDir()` 创建临时目录
  - 预览/执行引擎测试使用固定样例文件名与规则集
  - DB 相关测试使用 sqlite 内存库或临时文件

#### 覆盖重点

1. **预览计算引擎**：命名规则、批次冲突、磁盘冲突、自动后缀
2. **执行引擎**：移动/复制正确性、进度计算、错误记录
3. **历史服务**：写入/查询/撤销逻辑
4. **配置服务**：CRUD 与排序
5. **路径安全**：只允许白名单根目录

### 前端单测

- **推荐框架**：Vitest + React Testing Library
- **分层策略**：
  - 组件层：UI 输入/输出与交互
  - 页面层：核心流程交互与状态变化
  - API 层：mock 方式验证请求/响应适配
- **测试数据**：
  - 使用 MSW 或简单 fetch mock 固定响应
  - 对列表/分页等用例提供稳定的 fixture

#### 覆盖重点

1. **文件列表/预览**：数据展示、冲突标识
2. **执行流程**：任务创建 → 进度显示 → 完成反馈
3. **历史列表与撤销**：分页、详情展示、撤销触发
4. **系统配置**：源目录/预设/目标根目录管理

## 代码评审策略

### Review 维度

1. **与方案一致性**：对照 backend_plan.md / frontend_plan.md 检查功能是否完整实现
2. **结构与职责**：层次划分是否清晰，避免 handler 过重
3. **错误处理与日志**：是否统一格式/日志记录齐全
4. **安全与边界**：路径校验与文件系统访问的安全性
5. **可测试性**：是否能被单测覆盖，是否便于注入依赖

### Review 输出

- 对每个模块记录：
  - 是否符合方案
  - 主要问题与建议
  - 是否需要补充测试或重构

## 交付物

1. 单测实现（新增测试文件与必要的测试配置）
2. Review 记录（问题清单与结论）
3. CSV 任务跟踪表（测试 + Review 任务）

## Review 模板（建议）

### 结构与一致性

- 是否符合方案文档的功能描述
- 模块职责是否清晰、是否存在重复逻辑
- API 路由/参数命名是否一致

### 质量与安全

- 错误处理是否统一、是否存在遗漏
- 日志是否包含关键信息且不泄漏敏感数据
- 路径/文件操作是否全部经过安全校验

### 可测试性

- 是否能在不依赖外部环境下执行测试
- 是否便于 mock（例如接口抽象、依赖注入）
