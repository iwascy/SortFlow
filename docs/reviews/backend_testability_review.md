# SortFlow 后端可测试性评审（P1）

## 范围
- service 层依赖注入与可替换性
- handler 层测试便利性
- 并发与时间依赖对测试的影响

## 现状观察
- service 层多使用 `gorm.DB` 与文件系统直接依赖，当前单测主要通过 sqlite 临时库与 `t.TempDir()` 完成。
- `PreviewEngine` 逻辑相对纯净，便于单测。
- `ExecutionEngine` 内部直接创建 `HistoryService`，并在 goroutine 中执行，测试需要等待异步完成并依赖 DB 状态。

## 主要问题
1. **依赖注入不足**：执行引擎直接实例化历史服务，无法替换为 mock；文件系统操作直接使用 `os`，不便于隔离。
2. **异步执行难以控制**：`Execute` 使用 goroutine，缺少可等待/可取消机制，测试需要轮询或依赖任务状态。
3. **缺少时间/随机性抽象**：`UndoExpiresAt` 与任务时间使用 `time.Now()`，测试中需要额外控制时间。

## 建议
- 为 `ExecutionEngine` 引入接口依赖（历史写入、文件操作、时钟），通过构造函数注入实现。
- 提供可选的同步执行路径（或返回可等待的结果），降低测试的并发复杂度。
- 在 handler 层引入更细粒度的 service 接口，减少对 GORM 的直接依赖，提升 mock 可行性。

## 结论
当前结构可完成基础单测，但执行引擎与历史服务存在耦合与异步复杂性，进一步抽象依赖可显著提高可测试性。
