# SortFlow 后端错误处理与日志评审（P1）

## 范围
- handler 层错误响应与状态码
- middleware 错误处理与日志输出
- service 层错误传播与上下文

## 现状观察
- handler 层多数直接 `c.JSON` 返回 `{"error": err.Error()}`，错误结构不统一且直接暴露内部错误信息。
- 已有 `ErrorHandler` 中间件，但当前 handler 基本不调用 `c.Error`，因此该中间件难以统一处理错误响应。
- 日志中间件记录请求基本信息，缺少错误级别/错误详情的结构化输出。

## 主要问题
1. **错误响应不统一**：错误结构与状态码分散在各个 handler 中，且返回内容直接使用 `err.Error()`，不利于前端识别与安全控制。
2. **错误上下文不足**：service 层返回错误时缺少统一包装或错误码，难以在 handler 层做细分处理。
3. **日志缺少错误关联**：日志仅记录请求完成信息，未包含错误详情/错误码/trace id，排障成本较高。

## 建议
- 定义统一错误响应结构（如 `code/message/details`），并在 handler 中统一处理。
- 引入错误类型或错误码（`errors.Is`/`errors.As` + 自定义错误），将业务错误与系统错误区分。
- 对错误响应避免直接暴露底层错误信息，改为可控 message，并在日志中保留详细堆栈/错误信息。
- 强化日志输出：在错误发生时记录结构化字段（错误码、请求参数、调用链标识等）。

## 结论
错误处理与日志能力已具备基础，但缺少统一规范与结构化输出，建议在后续版本补齐错误模型与统一处理链路。
