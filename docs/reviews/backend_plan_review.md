# SortFlow 后端实现对照评审（P0）

## 与方案一致性
- 目录结构与 `backend_plan.md` 描述基本一致（handler/service/model/dto/pkg）。  
- 预览引擎、执行引擎、配置服务、历史服务与方案中的模块对应关系明确。  
- 需要补充：`tests/` 目录未使用，当前测试主要放在 `internal/service` 包下（已在测试计划中记录）。  

## 结构与职责
- handler 层以路由与参数绑定为主，核心逻辑位于 service 层，层级划分清晰。  
- 执行引擎与历史服务耦合较紧（执行后直接写历史），可考虑通过接口抽象或事件回调解耦，便于测试与扩展。  

## 质量与安全
- 路径安全校验使用白名单根目录前缀匹配；已补充测试并修复前缀误判风险（根目录前缀误判）。  
- 错误处理以直接返回或记录日志为主，缺少统一错误结构与错误码定义，建议后续统一错误响应格式。  

## 可测试性
- service 层逻辑可通过 sqlite 内存库和临时文件目录完成单测。  
- handler 层接口可通过 `httptest` 进行轻量验证；已开始覆盖 `/preview`、`/history`。  
- 建议为 `ExecutionEngine` 与 `HistoryService` 增加接口抽象，降低对 GORM 的直接依赖，提升 mock 可行性。  

## 问题清单与建议
1. 路径校验需确保根目录边界匹配，避免 `/allowed` 与 `/allowed-extra` 误判（已修复并测试）。  
2. handler 层错误响应格式不统一，建议定义标准错误结构。  
3. 历史与执行的耦合点可重构为可注入的接口或事件系统，便于扩展与测试。
