

### 1. 核心架构变更

* **配置存储：** 放弃 `yaml` 文件。使用 **SQLite** 嵌入式数据库存储分类（Presets）、历史记录和重命名规则。这允许用户在 Web 界面随时增删改查。
* **状态管理：** 前端引入 **“预检 (Dry Run)”** 机制。任何参数的修改，都不会改变文件，而是实时计算并更新“预期结果”视图。

---

### 2. 界面布局：三段式实时工作台

为了实现“不要弹窗”且“实时展示效果”，界面底部将变成一个常驻的 **“变更预览栏” (Change Staging Bar)**。

#### **A. 顶部：设置与管理**

* **设置入口：** 点击齿轮图标进入设置页，进行分类（Preset）的增删改。
* **分类管理 (CRUD)：**
* 添加分类：输入“分类名称”（如：风景）、“目标子目录”（如：`/Travel`）、“默认前缀”（如：`TR_`）。
* 支持拖拽排序（决定按钮在主界面的显示顺序）。



#### **B. 中部：源文件流 (Source Stream)**

* 与之前设计类似，但增加交互深度。
* **单选模式：** 点击某张图，底部面板进入“精细编辑模式”。
* **多选模式：** 框选或 Shift 连选，底部面板进入“批量模式”。

#### **C. 底部：核心操作区 (The Action / Preview Deck)**

这是改动最大的地方，它取代了弹窗。它分为上下两层：

* **上层：参数配置层 (Builder)**
* **分类按钮组：** [风景] [人物] [文档] (来自数据库动态加载)。
* **重命名构造器 (Renaming Mixer)：** 一组可开关、可拖拽排序的“积木块”。
* `[ ] 前缀` (勾选后自动填入当前分类设定的前缀)
* `[ ] 原文件名` (点击后展开高级选项：全名 / 正则提取 / 手动高亮)
* `[ ] 时间戳` (Exif / 文件创建时间)
* `[ ] 自动序列` (当检测到文件名冲突或多选时，自动激活 `-01`, `-02`)




* **下层：实时预览层 (Live Preview)**
* **左侧：** 显示 "已选中 X 个文件"。
* **中间 (关键)：** 动态显示的 **Diff 视图**。
* `Source: /Inbox/DSC_0001.jpg`  **➜** `Target: /NAS/Photos/Travel/TR_DSC_0001-01.jpg`
* 如果多选，这里显示前 3 个文件的预览，并提示 "...等 15 个文件"。
* **状态指示灯：**
* 🟢 **OK:** 路径正常，无冲突。
* 🟡 **Rename:** 目标已存在文件，将自动添加后缀 `-01`。
* 🔴 **Error:** 目标目录不可写 / 磁盘已满。




* **右侧：** **[ 执行 (Execute) ]** 按钮。只有状态为绿/黄时才亮起。

### 3. 核心功能逻辑详解

#### 3.1 动态重命名策略 (The Mixer)

针对您提到的“混合策略”和“自由选择部分文字”，逻辑如下：

**场景 A：单文件精细处理**
当用户只选中一个文件 `IMG_2023_SHANGHAI_001.jpg` 时：

1. 用户勾选 **“原文件名”** 模块。
2. 界面下方会出现该文件名的 **“分词视图” (Token View)**。系统按 `_`, `-`, `     ` 自动切分文件名。
* 显示为：`[IMG]` `[2023]` `[SHANGHAI]` `[001]`


3. 用户鼠标点击 `[SHANGHAI]`。
4. 此时预览结果实时变为：`CategoryPrefix_SHANGHAI.jpg`。
5. 用户实现了“自由多选部分文字”。

**场景 B：多文件批量处理**
当用户选中 50 个文件时，手动点击分词不再现实。

1. 用户勾选 **“原文件名”**。
2. 系统默认行为是 **Keep All** (保留全名)。
3. 系统自动检测冲突，并在预览结果中为重复项追加 `-01`, `-02`。
4. **高级选项：** 提供简单的正则提取（例如：“保留后5位”、“去除扩展名”），应用到所有选中的文件。

#### 3.2 自动序列与防冲突

后端 Go 逻辑在生成 `DryRun` 结果时执行以下检查：

```go
// 伪代码逻辑
func GeneratePreview(files []File, rules RenameRules) []PreviewItem {
    nameMap := make(map[string]int) // 用于记录当前批次内的命名频率
    
    for _, file := range files {
        newName := BuildName(file, rules) // 构建基础名
        
        // 1. 检查批次内冲突
        if count, exists := nameMap[newName]; exists {
            nameMap[newName]++
            newName = AddSuffix(newName, count) // 添加 -01
        } else {
            nameMap[newName] = 1
        }

        // 2. 检查磁盘物理冲突 (目标文件夹是否已有该文件)
        if FileExists(TargetDir + newName) {
             // 再次调整后缀，直到不冲突
             newName = FindAvailableName(TargetDir, newName)
        }
        
        results = append(results, PreviewItem{Src: file.Path, Dst: newName})
    }
    return results
}

```

### 4. API 接口设计 (Go)

后端需要提供更加交互式的接口。

* **`GET /api/settings/presets`**: 获取所有分类配置。
* **`POST /api/settings/presets`**: 添加/修改分类。
* **`POST /api/preview` (核心)**:
* **Input:** `{ file_ids: ["A", "B"], preset_id: 1, rules: { use_prefix: true, keep_original: true } }`
* **Output:**
```json
{
  "status": "ok",
  "conflicts_resolved": 2,
  "previews": [
    { "src": "...", "dst": "/NAS/Travel/TR_DSC01.jpg", "status": "ready" },
    { "src": "...", "dst": "/NAS/Travel/TR_DSC01-01.jpg", "status": "auto-renamed" }
  ]
}

```


* 前端每次改选项，就防抖 (Debounce) 调用一次这个接口，获取最新的预览。


* **`POST /api/execute`**:
* **Input:** 同上（也就是确认执行当前的预览方案）。
* **Output:** 任务 ID（用于前端显示进度条）。

### 5. 开发路线建议

1. **第一步 (Backend - Go):**
* 实现 SQLite 存储 Preset。
* 实现 `NameBuilder` 逻辑（字符串拼接、冲突检测）。
* **关键点：** 写好单元测试，模拟各种文件名冲突的情况，确保 `-01`, `-02` 生成逻辑无误。


2. **第二步 (Frontend - UI):**
* 搭建“底部操作栏”组件。
* 实现“点击文件名分词”的 UI 组件（类似 Tag Input）。
* 对接 `/api/preview`，确保点击任何开关，UI 上的目标路径都能立刻跳变。


3. **第三步 (Integration):**
* 联调真实文件移动。
* 增加进度条（如果是 NAS 上移动大文件，需要反馈）。
